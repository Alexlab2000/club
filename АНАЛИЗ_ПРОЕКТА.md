## 🔴 КРИТИЧЕСКАЯ ПРОБЛЕМА — переход на /home после подтверждения почты

**Симптом:** После клика по ссылке в письме — пользователь не попадает на /home.

**Причина найдена** в файле `app/auth/callback/route.ts`:

```ts
// ПРОБЛЕМА: redirect создаётся ДО того, как session установлена
const response = NextResponse.redirect(new URL(next, requestUrl.origin));

if (code) {
  // ...
  await supabase.auth.exchangeCodeForSession(code);
  // Куки сессии записываются в response, но...
}

return response; // ← redirect уже создан без куки!
```

**Что происходит:** браузер идёт на /home, но middleware видит пользователя
как незалогиненного (куки сессии не передались корректно), и отправляет обратно на /.

**Исправление** — переставить создание redirect ПОСЛЕ exchangeCodeForSession:

```ts
// app/auth/callback/route.ts — ПРАВИЛЬНАЯ ВЕРСИЯ:
export async function GET(request: NextRequest) {
  const requestUrl = new URL(request.url);
  const code = requestUrl.searchParams.get("code");
  const next = requestUrl.searchParams.get("next") ?? "/home";

  // Сначала — временный ответ без redirect (чтобы куки записались)
  const supabaseResponse = NextResponse.next({ request });

  if (code) {
    const supabase = createServerClient(
      process.env.NEXT_PUBLIC_SUPABASE_URL!,
      process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
      {
        cookies: {
          getAll() {
            return request.cookies.getAll();
          },
          setAll(cookiesToSet) {
            cookiesToSet.forEach(({ name, value, options }) =>
              supabaseResponse.cookies.set(name, value, options)
            );
          },
        },
      }
    );

    await supabase.auth.exchangeCodeForSession(code);
  }

  // Теперь redirect — ПОСЛЕ того как куки установлены
  const redirectUrl = new URL(next, requestUrl.origin);
  const response = NextResponse.redirect(redirectUrl);

  // Копируем все куки из supabaseResponse в redirect-ответ
  supabaseResponse.cookies.getAll().forEach((cookie) => {
    response.cookies.set(cookie.name, cookie.value, cookie);
  });

  return response;
}
```

## 🔴 КРИТИЧЕСКАЯ ПРОБЛЕМА — двойная проверка auth в /home

**Файлы:** `app/home/layout.tsx` + `app/home/page.tsx`

Оба файла делают `supabase.auth.getUser()` при каждой загрузке страницы.
Это два отдельных запроса к Supabase на один маршрут.

**Плюс:** middleware.ts тоже проверяет пользователя для каждого запроса.

Итого: **3 проверки** вместо одной. Это замедляет загрузку и создаёт 
состояние гонки (race condition) — layout может показать "Проверка доступа..."
пока page.tsx уже рендерится.

**Исправление:** убрать проверку из `home/layout.tsx` целиком — middleware 
уже защищает маршрут. Layout должен просто рендерить children:

```ts
// app/home/layout.tsx — УПРОЩЁННАЯ ПРАВИЛЬНАЯ ВЕРСИЯ:
export default function HomeLayout({ children }: { children: React.ReactNode }) {
  // Middleware уже проверил пользователя — здесь ничего не нужно
  return <>{children}</>;
}
```

## 🟡 СЕРЬЁЗНАЯ ПРОБЛЕМА — logout работает ненадёжно

**Файл:** `app/home/page.tsx`, функция handleLogout:

```ts
async function handleLogout() {
  await fetch("/api/logout", { method: "POST" });
  router.replace("/");
  router.refresh();
}
```

**Проблема:** `router.replace("/")` вызывается сразу после fetch, не дожидаясь
гарантии что сессия удалена на сервере. Если fetch завершится с ошибкой — 
пользователь будет думать что вышел, но сессия останется активной.

**Исправление:**
```ts
async function handleLogout() {
  try {
    const res = await fetch("/api/logout", { method: "POST" });
    if (!res.ok) throw new Error("logout failed");
  } catch {
    // Даже при ошибке — удаляем локальную сессию
    const supabase = createClient();
    await supabase.auth.signOut();
  }
  // Полная перезагрузка страницы гарантирует сброс всех состояний
  window.location.href = "/";
}
```

## 🟡 СЕРЬЁЗНАЯ ПРОБЛЕМА — check-answer: лимит 3 попытки, но в schema — 5

**Файл:** `app/api/check-answer/route.ts`:
```ts
const MAX_ATTEMPTS_PER_DAY = 3;  // ← в коде 3
```

**Файл:** `supabase/schema.sql`:
```sql
attempts_count INTEGER NOT NULL DEFAULT 0 CHECK (attempts_count BETWEEN 0 AND 5)  -- ← в базе 5
```

Несоответствие: база принимает до 5, код разрешает только 3.
Это не сломает сайт, но создаёт путаницу при поддержке.

**Исправление:** выровнять — выбрать одно число и поставить везде одинаково.

## 🟡 АРХИТЕКТУРНАЯ ПРОБЛЕМА — sessionKey в check-answer ненадёжен

**Файл:** `app/register/page.tsx`:
```ts
const sessionKey = useRef(
  typeof crypto !== "undefined"
    ? crypto.randomUUID()
    : Math.random().toString(36)
);
```

**Проблема:** sessionKey генерируется при каждом рендере страницы.
Если страница перезагрузилась — новый key, лимит попыток сбрасывается.
Пользователь может обойти лимит просто перезагрузив страницу.

**Исправление:** хранить sessionKey в sessionStorage браузера:
```ts
const sessionKey = useRef(() => {
  const stored = sessionStorage.getItem("reg_session_key");
  if (stored) return stored;
  const key = crypto.randomUUID();
  sessionStorage.setItem("reg_session_key", key);
  return key;
});
```

## 🟠 ПРЕДУПРЕЖДЕНИЯ — качество кода

### 1. Опечатка в данных клуба
**Файл:** `lib/club-content.ts`, группа id=1:
```ts
{ id: 1, title: "книги", icon: "🏛️", slug: "philosophy", ... }
```
Название "книги" написано строчными буквами, все остальные — с заглавной.
Иконка 🏛️ не соответствует теме "книги". Это видно пользователям на главной.

### 2. Телеграм-ссылки в публичном коде
**Файл:** `lib/club-content.ts`:
```ts
"1-1": [
  { id: 1, title: "Гл. 1-2 (гл.)", href: "https://t.me/c/1558317824/696" },
  ...
```
Ссылки на закрытый Телеграм-канал хранятся в исходном коде.
Любой кто получит доступ к коду (GitHub, хостинг) увидит эти ссылки.
Лучше хранить их в базе данных Supabase с RLS-защитой.

### 3. dynamic = "force-dynamic" на нескольких страницах
Добавлено в `page.tsx` и `register/page.tsx`. Это отключает кэширование
Next.js на этих маршрутах — каждый запрос будет рендериться заново.
Для страницы входа это излишне. Нужно только там где данные часто меняются.

### 4. Дублирование CookieToSet типа
Один и тот же тип `CookieToSet` определён в 4 местах:
- middleware.ts
- auth/callback/route.ts  
- api/login/route.ts
- api/logout/route.ts

Нужно вынести в `lib/supabase/types.ts` и импортировать.

### 5. next.config.js пустой
```js
const nextConfig = {};
```
Нет настроек безопасности: заголовков CSP, X-Frame-Options и т.д.

## 🔒 БЕЗОПАСНОСТЬ — общая оценка

### Хорошо сделано ✅
- RLS включён на всех таблицах базы
- question_answers доступны только через service_role (не anon)
- daily_question_attempts — тоже только service_role
- SUPABASE_SERVICE_ROLE_KEY не в NEXT_PUBLIC_ (не попадёт в браузер)
- middleware защищает /dashboard
- normalizeAnswer хорошо защищает от вариаций ввода

### Нужно добавить ⚠️

**Заголовки безопасности** в `next.config.js`:
```js
const nextConfig = {
  async headers() {
    return [
      {
        source: "/(.*)",
        headers: [
          { key: "X-Frame-Options", value: "DENY" },
          { key: "X-Content-Type-Options", value: "nosniff" },
          { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
          { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
        ],
      },
    ];
  },
};
```

**Middleware не защищает /home** — только /dashboard и /:
```ts
// В middleware.ts сейчас:
if (!user && pathname.startsWith("/dashboard")) { ... }   // ← защита /dashboard
if (user && (pathname === "/" || pathname === "/register")) { ... } // ← редирект
// Нет защиты для /home !
```

Добавить строку:
```ts
if (!user && (pathname.startsWith("/dashboard") || pathname.startsWith("/home"))) {
  return NextResponse.redirect(new URL("/", request.url));
}
```
